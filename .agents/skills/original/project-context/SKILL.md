# プロジェクトコンテキストの参照ルール

新しいタスクを開始する際や、実装方針に迷った場合、あるいはこのプロジェクト固有の役割分担を確認する必要がある場合は、必ずこのファイルを参照して現在のタスクに適用してください。
もしこのファイルの内容とユーザーの指示に矛盾がある場合は、必ずユーザーに確認してください。

# リポジトリガイドライン

## C++ コーディングスタイルガイドライン

* モダンな C++20 を使用してください。
* パブリック API の Doxygen ドキュメントストリングは、ヘッダーファイル内に英語で記述し、アメリカ英語と能動態を使用してください。

## レビューガイドライン

* レビューコメントは自然な日本語で記述し、英語は識別子や既存のコメントを引用する場合にのみ使用してください。
* コメントは簡潔かつ直接的で、プロフェッショナルなものとし、指示書への言及は避けてください。
* 該当する場合は、英語の文法やタイポ（誤植）をチェックし、このファイルのガイドラインに照らし合わせてコーディングスタイルをレビューしてください。

## コミットメッセージのガイドライン

* 各コミットはアトミック（不可分）に保ってください：1つのコミットには1つの論理的な変更のみを含めます。
* 次のフォーマットを使用してください：
  ```
  <emoji> <description>

  <details>
  ```

`<description>` のルール：

* 簡潔な英語で記述し、動詞の原形から始め、最初の文字は大文字にしてください。
* 変更内容とその目的を約50文字で説明し、コードの識別子はバッククォートで囲んでください。

`<emoji>` のルール：

* コミットの主な意図と一致する Gitmoji を先頭に1つだけ使用してください。
* マッピング： 🐛 `fix`, ✨ `feat`, 🛠️ `build`, 💚 `ci`, 📝 `docs`, 🎨 `style`, ♻️ `refactor`, ⚡️ `perf`, ✅ `test`, 🩹 `chore`, 🚧 `wip`, 🤖 `ai`

`<details>` のルール：

* 変更内容を箇条書きで具体的に記述する
* 1つの変更であっても目的が異なる場合は別の行に分けて記述する
* 本質的でない変更（空行の追加など）は最後に記述する

## ビルドとテスト

* CMake コンフィギュレーション： `cmake -S . -B build -DCMAKE_TOOLCHAIN_FILE=externals/vcpkg/scripts/buildsystems/vcpkg.cmake -G "Ninja Multi-Config"`
* ルートディレクトリでのコンパイルコマンドの生成： `cmake --build build --config Debug --target gen_compile_commands`
* ビルド： `cmake --build build --config Debug --clean-first`
* テストの実行： `ctest --test-dir build -C Debug --output-on-failure`

## フォーマットとリンターのチェック

* コミットする前に、ローカルでフォーマットとリンターによるチェックを実行してください。
* 可能であれば、`/usr/bin` にある最新の `clang-format` および `clang-tidy` を使用してください。
* 「ビルドとテスト」セクションに従ってコンパイルコマンドを生成した後、変更された C++ ファイルに対して、リポジトリの設定で `clang-format` と `clang-tidy` を実行してください。
