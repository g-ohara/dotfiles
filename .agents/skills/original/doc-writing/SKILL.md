# ドキュメント作成ルール

## 概要

このルールは、エージェントが Markdown 形式のドキュメントを作成・修正する際に常に従うべき条件を定義する。
すべてのタスクで適用される。

## 文体ルール

### 1. 常体を使用する

* `である`、`できない`、`ならない` などの常体を使用する
* `です`、`ます` などの敬体は使用しない
* コードブロック内のコメントは適宜（例: `// エラー！一時オブジェクトへの参照は保存できない`）

### 2. 句点を省略する

* 箇条書きの項目末尾に句点（`.`）を付けない
* 段落の末尾には句点を付けても可

### 3. 日本語と英語の使い分け

* 説明文は日本語で記述する
* コード識別子、型名、関数名などは英語のままにする
* 技術用語は必要に応じて日本語訳を併記する（例: `ハフマン木（Huffman tree）`）

## フォーマットルール

### 1. テストケースのトグル表示

* `<details>` / `<summary>` を使用してテストケースをトグル可能にする
* `<summary>` にはセクションの見出しをそのまま記載する（見出しマーク `####` は含まない）
* 各テストケースごとに `<details>` を作成せず、**セクション単位** でまとめる

```markdown
<details>
<summary>正常系：基本フォーマット</summary>

| 入力 | 期待される結果 |
|------|---------------|
| `"aaaaabbbbcc"` | ... |
| `"hello"` | ... |

</details>
```

### 2. テーブルのフォーマット

* 表の区切り文字と内容はスペース 1 つ空ける
* 列揃えは行儀よく行う

```markdown
| メンバ/メソッド | 説明 |
|-----------------|------|
| `char ch_` | 葉ノードの場合、対応する文字を保持 |
```

### 3. コードブロック

* C++ コードには `cpp` ラベルを付ける
* 重要な箇所は適切なインデントを保つ
* 説明文の中に埋め込む場合は、簡潔なものに限る

### 4. 見出しの階層

* `##` をトップレベルとして使用する
* 必要に応じて `###`、`####` と階層を下げる
* 番号付きの見出し（`1.`、`2.` など）は使用しない

## 自動適用チェックリスト

ドキュメントを作成・修正する際、以下の項目を自動的に確認・適用する：

- [ ] 文体が常体になっているか（`です/ます` → `である`）
- [ ] 箇条書きの末尾に句点がないか
- [ ] テストケースが `<details>` / `<summary>` で囲まれているか
- [ ] `<summary>` に見出しマークが含まれていないか
- [ ] テストケース見出しの番号が削除されているか
- [ ] 環境非依存の型（`int32_t`、`uint64_t`）を使用しているか
- [ ] コンストラクタに `explicit` が付いているか
- [ ] メンバ変数の末尾に `_` が付いているか
- [ ] ゲッタに `Get` プレフィックスが付いているか
- [ ] 例外条件が明記されているか
- [ ] EXTRA 機能の実装保留が記載されているか（該当する場合）

## ワークフロー

1. ユーザーからドキュメント作成・改善を依頼されたら、既存のドキュメントを参考にする
2. 上記のルールを自動で適用する
3. ドラフト版を出力し、ユーザーに確認を求める
4. ユーザーからのフィードバックに基づいて修正を行う
5. 最終版を確定する

## 注意事項

* ユーザーが特定の指示をした場合、その指示を優先する
* 不明確な点はユーザーに確認する
* 既存のドキュメントとの整合性を保つ
